# 验收补充要求 - 聚合器联动细化版

## 1. 适用范围
1. 本文用于补充《验收报告 - 金融数据源集成.md》
2. 目标是完成“爬虫聚合器对下游可消费”的关键验收
3. 仅当本文 P0 项全部通过，方可签署“通过/有条件通过”

## 2. P0 必验项（必须全部通过）

### P0-1 批次交付协议验收
1. 检查目录：`exports/<batch_id>/`
2. 必须存在文件：`events.jsonl`、`manifest.json`、`_SUCCESS`
3. 通过标准：
- 无 `_SUCCESS` 的批次必须被视为失败批次
- 最新可消费批次必须同时具备上述 3 个文件
4. 证据：目录截图或文件清单输出

建议命令：
```powershell
Get-ChildItem -LiteralPath .\exports -Directory | Sort-Object Name -Descending | Select-Object -First 3 Name
Get-ChildItem -LiteralPath .\exports\<batch_id> | Select-Object Name,Length,LastWriteTime
```

### P0-2 Schema 与字段质量验收
1. 抽检 `events.jsonl` 至少 1000 条（不足则全量）
2. 必填字段：
- `event_id, source, source_id, title, published_at, url, event_type, ingested_at`
3. 时间格式：ISO8601 且带时区（例如 `+08:00`）
4. 通过标准：
- 必填字段非空率 >= 99%
- 时间格式正确率 >= 99%
5. 证据：字段统计表 + 异常样本（最多 20 条）

### P0-3 幂等性验收
1. 在相同输入窗口连续执行导出两次
2. 对比两次导出新增量
3. 通过标准：
- 第二次新增量接近 0（仅允许真实新抓取数据）
- 相同事件 `event_id` 不变化
4. 证据：两次批次 diff 统计

### P0-4 Watermark 断点续跑验收
1. 验证生产位点（producer watermark）按 source 推进
2. 人为中断后重跑，验证不会重复导出旧事件
3. 多消费者分别维护消费位点（consumer watermark）
4. 通过标准：
- 重跑不重复
- 不同 consumer 互不干扰
5. 证据：watermark 文件前后对比

### P0-5 完整性与校验验收
1. 校验 `manifest.record_count` 与 `events.jsonl` 行数一致
2. 校验 `manifest.checksum` 与文件实际 sha256 一致
3. 通过标准：两项均一致
4. 证据：校验命令输出

建议命令：
```powershell
(Get-Content -LiteralPath .\exports\<batch_id>\events.jsonl).Count
Get-FileHash -LiteralPath .\exports\<batch_id>\events.jsonl -Algorithm SHA256
Get-Content -LiteralPath .\exports\<batch_id>\manifest.json
```

### P0-6 可追溯性验收
1. 随机抽样不少于 50 条事件
2. 检查可回溯字段：`url` 或 `file_path`
3. 通过标准：
- 可回溯成功率 >= 98%
- 抽样中不得出现不可定位来源的事件
4. 证据：抽样明细表

## 3. P1 建议验收项（建议通过）
1. 单源失败隔离：某源故障时，其它源批次仍可导出
2. 性能基线：1 万条事件导出耗时 < 5 分钟（单机）
3. 监控可用：可查询最近任务状态、失败原因、分源产量
4. 噪音控制：导航类/模板类标题占比 <= 5%

## 5. 验收结论判定规则
1. `通过`：P0 全部通过，P1 不影响上线
2. `有条件通过`：P0 仅允许 1 项轻微偏差，且有明确修复计划（<= 3 个工作日）
3. `不通过`：任意 P0 失败，或证据不足无法复核
